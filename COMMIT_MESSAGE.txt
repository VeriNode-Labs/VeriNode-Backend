Fix: Reputation system race condition with atomic operations

Problem:
--------
Race condition in reputation scoring where concurrent reward (+10) and 
slashing (-500) events could result in the slashing being entirely lost.

Example:
  Initial score: 750
  Concurrent: Reward +10, Slashing -500
  Bug: Final score could be 760 (slashing lost) ❌
  
This was a write-skew anomaly where the last write would overwrite the 
previous operation, causing critical slashing penalties to be lost.

Solution:
---------
1. Atomic UPDATE operations for rewards (no read-write gap)
2. Row-level locking with SELECT FOR UPDATE NOWAIT for slashings
3. Priority enforcement ensuring slashing is never lost
4. Complete audit trail in reputation_events table
5. Score constraints enforced at database level

Implementation:
---------------
- Database schema: 005_reputation_schema.sql
  - reputations table with score constraints [-1000, 1000]
  - reputation_events audit log table
  - Indexes for performance
  - Slash version tracking

- Store layer: src/reputation/store.ts
  - Atomic reward operations (no locking needed)
  - Serialized slashing operations (row-level locks)
  - Event logging for all operations
  - Transaction management

- Service layer: src/reputation/scoreService.ts
  - Business logic with reward/slashing operations
  - Configuration constants (REWARD_DELTA: 10, SLASHING_DELTA: -500)
  - Logging and monitoring integration
  - Invariant verification
